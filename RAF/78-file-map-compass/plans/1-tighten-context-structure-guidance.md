# Task: Tighten Context Structure Guidance

## Objective
Make RAF consistently describe shared project context with a `## Project Files` section that lists concrete file paths and with a `## Key Decisions` section reserved for durable domain-level decisions.

## Requirements
- Update the shared planning/amend prompt contract so it says `## Project Files` instead of `## Relevant Files`.
- Require `## Project Files` entries to use concrete file paths rather than bare directories or globs, and instruct the executor to inspect each file if it becomes relevant.
- Narrow the `## Key Decisions` guidance to durable project, product, code, API, and architecture decisions; exclude planning tactics, task organization, and other execution-process notes.
- Preserve the `## Key Decisions` heading itself so downstream consumers that read that section name continue to work.
- Keep README guidance, prompt-viewer sample data, and regression tests aligned with the new wording so the contract does not drift across surfaces.
- If stale tests still describe RAF as auto-generating richer `context.md` structures, reconcile them with the current read-only `context.md` lifecycle instead of reintroducing removed behavior.

## Key Decisions
- This project changes the planning contract, not RAF's runtime `context.md` ownership model: `context.md` remains planner/executor-maintained rather than auto-generated by RAF.
- The rename to `## Project Files` should come with stricter content rules, not just terminology replacement, so future agents stop listing folders like `plans/` and `outcomes/`.
- `## Key Decisions` must capture durable domain decisions only; process-management notes belong in other sections or should be omitted.

## Acceptance Criteria
- [ ] Planning and amend prompt text both instruct agents to use `## Project Files` and to list concrete file paths with "inspect if relevant" style guidance.
- [ ] Planning and amend prompt text both define `## Key Decisions` as durable project/product/code/API/architecture decisions and explicitly exclude planning or task-organization tactics.
- [ ] README and prompt-viewer fixtures no longer describe `## Relevant Files` or directory-only examples for shared project context.
- [ ] Regression coverage asserts the new section name and semantics without requiring RAF to resurrect removed `context.md` generation behavior.

## Context
The current shared prompt rules in `src/prompts/shared.ts` still describe `## Relevant Files` using examples like `plans/` and `outcomes/`, and the plan template still defines `## Key Decisions` broadly enough that process-oriented notes have been treated as valid plan content. Those instructions flow into both planning and amend prompts, are reflected in README guidance, and are mirrored by prompt-viewer sample data and prompt tests.

## Implementation Steps
1. Update the shared wording in `src/prompts/shared.ts` so the project-context rules and plan template describe `## Project Files` with file-path-only guidance and redefine the purpose of `## Key Decisions`.
2. Validate the rendered planning and amend prompts in `src/prompts/planning.ts` and `src/prompts/amend.ts`, adjusting any flow-specific wording that still mentions `## Relevant Files` or broad planning-time decisions.
3. Align user-facing references in `README.md` and prompt-viewer fixtures (`prompt-viewer/generate.mjs`, regenerated `prompt-viewer/prompts-data.js` if needed) with the new structure and examples.
4. Update prompt regression tests to assert the renamed section and stricter semantics, and reconcile any stale `project-context` tests only as needed to match RAF's current read-only `context.md` lifecycle.
5. Run the focused test set for touched prompt and context files, plus any fixture regeneration step needed by the prompt viewer.

## Risks & Mitigations
- Terminology drift across surfaces: update the shared prompt source first, then verify README, prompt-viewer fixtures, and tests against the rendered output rather than editing them independently.
- Over-correcting by changing downstream parsers: keep the `## Key Decisions` header name unchanged and limit the work to guidance, examples, and tests unless an actual consumer breaks.
- Stale context-builder tests obscuring the real request: treat them as compatibility cleanup only if they block or misrepresent the new contract; do not rebuild removed generation code to satisfy them.
