# Add Always-On Project context.md With Minimal Prompt Duplication

  ## Summary

  Add a single generated context.md file to every RAF project and keep it updated from both planning and execution
  flows. This file becomes the default compact project context for plan --amend and do.

  The design is always-on and requires no config. Prompts should stay lean:

  - inject context.md inline
  - provide only the extra file paths that are directly needed for the current step
  - avoid repeating information already present in context.md

  ## Key Changes

  - Add a generated project artifact at:
      - ${projectPath}/context.md
  - Implement a deterministic context builder that composes one bounded markdown summary from:
      - input.md
      - decisions.md
      - current task status derived from plans/outcomes
      - completed outcome files
  - Refresh context.md in raf plan after plans are created and before planning artifacts are committed.
  - Refresh context.md in raf plan --amend after the session finishes and before planning artifacts are committed.
  - Refresh context.md in raf do after each outcome write so the next task sees updated project-wide progress.
  - Include context.md in planning-artifact commits for plan/amend flows.

  ## context.md Content

  Use one stable markdown structure:

  - # Project Context
  - ## Goal
  - ## Key Decisions
  - ## Current State
  - ## Completed Work
  - ## Pending Work
  - ## Source Files

  ## Source Files should be the canonical place for shared project references:

  - input.md
  - decisions.md
  - plans/
  - outcomes/
  - selected recent outcome files when useful

  ## Completed Work should contain short entries for completed tasks:

  - task ID
  - task name
  - compact outcome summary

  Prefer extracting each outcome’s ## Summary section when present; otherwise use first meaningful bounded content.

  ## Prompt Changes

  - Update raf do prompt generation in src/prompts/execution.ts to inline context.md near the top.
  - Keep only task-specific paths outside context.md:
      - current plan path
      - current outcome file path
      - previous attempt outcome path on retry
  - Remove repeated generic project-file references from the execution prompt when that information is already present
    in context.md.
  - Keep dependency context in the execution prompt, but avoid re-listing shared project paths there.
  - Update raf plan --amend prompt generation in src/prompts/amend.ts to inline context.md and keep the existing
    protected/modifiable task list.
  - Avoid duplicating project-level file-location guidance in the amend prompt if context.md already covers it.

  ## Implementation Notes

  - Add a dedicated context builder module rather than scattering summarization logic across commands.
  - Reuse the existing outcome summarization approach from src/prompts/execution.ts where possible.
  - The builder should derive task status from existing state machinery instead of reparsing markdown ad hoc.
  - context.md is generated by RAF and treated as replaceable output, not user-authored content.

  ## Test Plan

  - Add unit tests for context generation:
      - builds context.md from input, decisions, state, and outcomes
      - includes source file references once
      - prefers outcome ## Summary sections
      - stays compact and does not duplicate sections
  - Update plan/amend flow tests to verify context.md is created or refreshed and committed with planning artifacts.
  - Update execution prompt tests to verify:
      - context.md content is included
      - current plan/outcome paths are still present
      - shared project file references are not redundantly repeated outside context.md
      - dependency and retry context still appear correctly
  - Add a do progression test ensuring task N updates context.md and task N+1 receives the refreshed context.

  ## Assumptions

  - context.md is always generated and always used.
  - The current task plan remains the primary task contract.
  - context.md is the single source of project-level summary and shared file references, so prompts should not restate
    that material unless a path is directly required for the current task.
